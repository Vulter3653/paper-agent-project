# Benchmark Standard v3: Implementation-Ready Metric Specification

## 1. Purpose

Benchmark Standard v3 converts the 30-metric layered architecture from v2 into implementation-ready metric specifications. Each metric receives a definition, formula or rule, required inputs, output field, evidence level, pass condition, failure meaning, and dashboard display guidance.

Benchmark Standard v3는 v2에서 구조화한 30개 metric을 실제 구현 가능한 평가 명세로 전환한다. 각 지표는 정의, 계산식 또는 판정 규칙, 필요한 입력, 출력 필드, 증거 수준, 통과 기준, 실패 의미, 대시보드 표시 방식을 갖는다.

## 2. Relationship Between v2 and v3

Benchmark Standard v2 defines the automated benchmark architecture and metric layers. Benchmark Standard v3 does not introduce new benchmark claims. It makes the v2 metric architecture implementation-ready by assigning 10 specification fields to each of the 30 metrics.

v2 defined the architecture. v3 defines how each metric should be computed, interpreted, displayed, and used for claim-boundary control.

v2는 평가 아키텍처와 metric layer를 정의한다. v3는 새로운 성능 주장을 추가하지 않는다. v3는 v2의 30개 지표 각각에 10개 specification field를 부여하여 구현 가능성을 높이는 명세화 단계이다.

## 3. Metric Specification Field Definitions

| Field | Meaning |
| --- | --- |
| metric_name | Stable machine-readable metric identifier. |
| metric_layer | One of the six v2 layers. |
| definition | Plain-language definition of what the metric measures. |
| formula_or_rule | Mathematical formula, deterministic rule, or LLM judge scoring rule. |
| input_required | Required input files or fields. |
| output_field | Output column or JSON field name to be generated by future scripts. |
| evidence_level | Gate, primary, deterministic validity, diagnostic, semantic proxy, robustness, or claim-safety evidence. |
| pass_condition | Threshold or logical condition required for pass status. |
| failure_meaning | What a low score or failed gate means. |
| dashboard_use | How the metric should be displayed in the dashboard. |

## 4. Metric Evidence Hierarchy

Evidence hierarchy:

1. Foundation and reproducibility gates
2. Schema and metadata readiness
3. Deterministic DOI, paper existence, journal, and OA validity checks
4. Gold matching and retrieval ranking metrics
5. Fixed LLM-as-a-judge semantic proxy metrics
6. Robustness and risk metrics

LLM-as-a-judge scores cannot override deterministic DOI, metadata, paper existence, journal policy, or gold matching failures.

LLM-as-a-judge 점수는 DOI, 메타데이터, 논문 존재 여부, 저널 정책, gold matching 실패를 대체하거나 무효화할 수 없다.

## 11. Dashboard Mapping Rules

- **Foundation Gates**: Displayed as "READY/NOT READY" status badges.
- **Accuracy Layer**: Displayed as comparative bar charts (Rule-based vs Single LLM vs Agent).
- **Validity/Integrity**: Displayed as "Trust Gauges" (Percentage scale).
- **Risk Layer**: Displayed in an "Agent Ops" monitoring panel.

## 12. Claim Boundary Rules

- **Strict Mode**: Only metrics from Layers 1-4 that have passed "automated_validation" can be used for performance claims.
- **Experimental Mode**: Layer 5 scores can be used to describe semantic potential but must be labeled as "LLM Proxy".
- **Evidence Labeling**: All metrics must be tagged with the evidence level (e.g., `[Gate]`, `[Primary]`, `[Proxy]`).
